NestJS 백엔드 변경 코드에 대해 플로우 기반 QA 리뷰를 수행합니다.
모든 판단은 근거 기반(코드 직접 확인)으로만 진행합니다. 추측/추론 금지.
프론트엔드 프로젝트 경로: `../next-bun`

## 1단계: 변경 파일 파악

`git diff --name-only`로 변경 파일 목록을 확인합니다.
변경 파일에서 추가/수정된 함수, API, 타입을 식별합니다.

## 2단계: grep 전수 확인

변경된 함수/API/타입 이름으로 **백엔드 + 프론트엔드** 프로젝트 전체 grep 실행.
- 동일 함수가 여러 곳에서 호출되면 중복/불필요 여부 교차 비교
- 파일 부분 읽기(offset/limit)만으로 "전체 정상" 판단 금지

## 3단계: 플로우 리스트업

변경 내용을 유저 액션 기준 플로우로 분류합니다.
각 플로우: `유저 액션 → 프론트 핸들러 → API 호출 → 백엔드 Controller → Service → DB → 소켓 emit`

## 4단계: 플로우별 코드 추적 검증

각 플로우마다:
1. **중복 호출**: 동일 API/함수가 불필요하게 여러 번 호출되지 않는지
2. **소켓 이벤트**: emit 횟수, 페이로드 필드, self-event 필터링, 토스트 중복 없는지
3. **타입 일치**: 엔티티 → 쿼리 select → 서비스 매핑 → DTO → 프론트 타입 → 소켓 페이로드 체인 전체 1:1 대조
4. **필드 누락**: 새로 추가한 필드가 위 체인 전체에 포함되는지
5. **기능 정합성**: 의도한 데이터만 변경되고, 불필요한 부수효과 없는지

## 5단계: Swagger 확인

변경된 엔드포인트의 Swagger 데코레이터 확인:
- `@ApiProperty` (DTO 필드)
- `@ApiQuery`, `@ApiParam` (쿼리/경로 파라미터)
- `@ApiResponse` (응답 상태코드)
- `@ApiOperation` (엔드포인트 설명)

## 6단계: 중복 코드 / 리팩토링

- 동일 패턴이 3곳 이상 반복되면 공통 함수 추출
- 오버엔지니어링 금지

## 7단계: 빌드 검증

- 백엔드: `pnpm run build`
- 프론트: `cd ../next-bun && npx next build`
